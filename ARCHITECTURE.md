# Architecture & Design

## System Overview

Prism is a **multi-agent feedback intelligence system** built on LangGraph. It transforms unstructured team feedback into actionable work items through a pipeline of specialized AI agents.

```
┌────────────┐
│   Intake   │  Raw messages from Discord, Slack, or manual input
│   Queue    │
└─────┬──────┘
      │
      ▼
┌────────────────────────────────────────────────────────────┐
│  Agent Pipeline (LangGraph)                                │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 1. CLASSIFIER                                       │ │
│  │    Input: Raw text                                 │ │
│  │    Output: {lane, title, priority, summary}        │ │
│  └──────────────────────────────────────────────────────┘ │
│  ▼                                                        │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 2. ISSUE INTERPRETER (optional, interactive)        │ │
│  │    Input: Raw feedback + user refinement           │ │
│  │    Output: Structured issue card                   │ │
│  │            - Acceptance criteria                    │ │
│  │            - Edge cases                             │ │
│  │            - Clarification needs                    │ │
│  └──────────────────────────────────────────────────────┘ │
│  ▼                                                        │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 3. IMPLEMENTATION ARCHITECT (optional, interactive) │ │
│  │    Input: Issue card                               │ │
│  │    Output: Implementation plan                      │ │
│  │            - Step-by-step tasks                     │ │
│  │            - Testing checklist                      │ │
│  │            - Risk assessment                        │ │
│  └──────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────┘
      │
      ▼
┌────────────┐
│  Kanban    │  4 lanes: Issues, Features, Ideas, Marketing
│   Board    │  Status lifecycle per lane
└────────────┘
```

## Core Components

### 1. **Intake & Deduplication** (`queue_store.py`)

Messages arrive from three sources:
- Discord (daemon bot, WebSocket)
- Slack (daemon bot, Socket Mode)  
- Manual (REST API or UI paste)

**Deduplication**: MD5 hash of `{source}:{content}` ensures no duplicate processing.

**Status flow**: `pending` → `classified` → (optional: human review + AI refinement)

### 2. **Agents** (`agent.py`, LangGraph)

All agents are **stateless node functions** in a directed graph.

#### Agent 1: Classifier
- **Purpose**: Quick categorization of raw feedback
- **Prompt**: JSON classification prompt (see `CLASSIFIER_PROMPT`)
- **Output**: 
  ```json
  {
    "lane": "issue",
    "title": "Mobile login broken",
    "priority": "critical",
    "summary": "Users can't authenticate on iOS"
  }
  ```
- **Latency**: 1-3 seconds (depends on LLM provider)
- **Cost**: ~$0.001 per call (OpenAI GPT-4o)

#### Agent 2: Issue Interpreter
- **Purpose**: Deep analysis of messy feedback into structured cards
- **Prompt**: Instruction to extract acceptance criteria, edge cases, risks
- **Output**: Markdown-formatted issue card (see `ISSUE_INTERPRETER_PROMPT`)
- **Manual**?: Can be reviewed/edited before saving
- **Latency**: 3-8 seconds

#### Agent 3: Implementation Architect
- **Purpose**: Convert issue card into buildable technical plan
- **Prompt**: Technical planning prompt (see `IMPLEMENTATION_ARCHITECT_PROMPT`)
- **Output**: Markdown-formatted tech plan with steps, testing, risks
- **Manual**?: Can be reviewed/edited

### 3. **Database Layer** (`queue_store.py`, SQLite)

Three tables:

#### `messages`
```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY,
    source TEXT,              -- 'discord', 'slack', 'manual'
    content TEXT,             -- Raw message text
    channel TEXT,             -- Where it came from
    author TEXT,              -- User who said it
    status TEXT,              -- 'pending', 'classified', 'dismissed'
    msg_hash TEXT UNIQUE,     -- Dedup key
    lane TEXT,                -- 'issue', 'feature', 'idea', 'marketing'
    title TEXT,               -- Extracted title
    summary TEXT,             -- One-line summary
    priority TEXT,            -- 'critical', 'high', 'medium', 'low'
    item_status TEXT,         -- Status within lane (e.g., 'new', 'confirmed')
    issue_card TEXT,          -- Generated by Interpreter
    tech_plan TEXT,           -- Generated by Architect
    notes TEXT,               -- Team notes
    created_at DATETIME
);
```

#### `config`
```sql
CREATE TABLE config (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

Stores:
- LLM API keys (OpenAI, Groq)
- Bot tokens (Discord, Slack)
- Channel configs

#### `context_files`
```sql
CREATE TABLE context_files (
    id INTEGER,
    filename TEXT,
    content TEXT (max 60KB),  -- Knowledge base for agents
    file_type TEXT,
    created_at DATETIME
);
```

Allows users to upload docs/specs that get injected into agent prompts.

### 4. **UI** (`app.py`, Streamlit)

Single-page Streamlit app with custom CSS.

**Pages** (via sidebar radio):
1. **Inbox** — Pending messages
2. **Board** — Kanban (4 lanes)
3. **Item Details** — Full card for selected item
4. **Settings** — API keys, bot config, context files

**Key interactions**:
- Drag/drop messages to lanes (via buttons)
- One-click "Auto-Process" → triggers Classifier
- One-click "Generate Issue Card" → triggers Interpreter
- One-click "Generate Plan" → triggers Architect
- Edit any field inline

### 5. **Backend API** (`server.py`, FastAPI)

Optional REST backend (not required if using UI only).

**Key endpoints**:

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/inbox` | Get pending messages |
| POST | `/api/inbox` | Add message |
| DELETE | `/api/inbox/{id}` | Dismiss message |
| POST | `/api/process/{id}` | Auto-classify (Classifier agent) |
| GET | `/api/board?lane=issue` | Get board items by lane |
| POST | `/api/item/{id}/issue-card` | Generate issue card |
| POST | `/api/item/{id}/tech-plan` | Generate tech plan |

**Threading**: `ThreadPoolExecutor(max_workers=4)` for async LLM calls

### 6. **Integration Bots** (Discord & Slack)

#### Discord (`discord_bot.py`)
- Uses `discord.py` async client
- Runs in background daemon thread
- Watches channels configured in `settings → Discord Channels`
- Auto-adds messages to intake queue
- Matches by channel ID or name (configurable)

#### Slack (`slack_bot.py`)
- Uses `slack-sdk` Socket Mode
- Runs in background daemon thread
- No public URL needed (WebSocket-based)
- Watches channels configured in settings
- Resolves channel IDs to names for display

## LLM Providers

Support for multiple LLM backends:

| Provider | Key Format | Model | Speed | Cost |
|----------|-----------|-------|-------|------|
| OpenAI | `sk-...` | GPT-4o | 2-5s | ~$0.002 per call |
| Groq | `gsk_...` | Llama 3.3 70B | 0.5-1s | Free tier available |

**Provider detection**:
```python
def get_llm(api_key: str):
    if api_key.startswith("sk-"):
        return ChatOpenAI(model="gpt-4o", api_key=api_key)
    elif api_key.startswith("gsk_"):
        return ChatOpenAI(model="llama-3.3-70b-versatile", 
                         base_url="https://api.groq.com/openai/v1")
```

## Strengths & Design Decisions

### ✅ Why SQLite?
- ✨ Zero setup (no separate DB server)
- ✨ Perfect for demo/MVP
- ⚠️ Not suitable for >10 concurrent users
- 🔄 Easy to migrate to PostgreSQL later

### ✅ Why LangGraph?
- ✨ Composable agent nodes
- ✨ Easy to add more agents
- ✨ Type-safe state management (TypedDict)
- ✨ Built-in error handling & retries

### ✅ Why Streamlit?
- ✨ Fast iteration (no React/Vue boilerplate)
- ✨ Beautiful default components
- ✨ Custom CSS hooks for styling
- ✨ Built-in state management

### ✅ Why Socket Mode (Slack)?
- ✨ No public URL needed
- ✨ Great for testing behind firewalls
- ⚠️ Slightly higher latency than webhooks

## Limitations & Future Scalability

| Aspect | Current | Future |
|--------|---------|--------|
| Database | SQLite | PostgreSQL |
| Concurrency | Single-threaded DB | 10+ connections |
| LLM | External APIs only | Local LLM support (Ollama) |
| Auth | None | User/team accounts |
| Cache | None | Redis caching for LLM responses |
| Webhooks | Manual | Event-driven automation |
